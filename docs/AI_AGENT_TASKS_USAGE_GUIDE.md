# AI 에이전트 Task 정의서 활용 가이드

## 1. 개요
이 가이드는 `tasks/` 디렉토리 하위에 정의된 "AI 에이전틱 프로그래밍용 Task 정의서"를 에이전트가 어떻게 해석하고 실행해야 하는지를 설명합니다. 모든 Task 정의서는 **Human-readable Markdown** 설명과 **Machine-readable YAML** 블록으로 구성되어 있습니다.

## 2. Task 정의서 구조

### 2.1. 파일 포맷
각 Task 파일(`.md`)은 다음과 같은 표준 구조를 가집니다.

````markdown
# REQ-FUNC-XXX: Task 제목

## 1. 개요 및 목적
(사람이 읽기 위한 상세 설명, 설계 의도, 배경 지식)

## 2. 상세 요구사항
(입력, 출력, 처리 과정, 예외 처리 등에 대한 서술적 묘사)

... (기타 섹션) ...

## 3. Task Definition (YAML)
```yaml
task_id: "REQ-FUNC-XXX-BE-01"
type: "functional"
... (구조화된 데이터) ...
```
````

### 2.2. 주요 YAML 필드 정의


| 필드명 | 필수여부 | 설명 | 에이전트 활용 방법 |
|---|---|---|---|
| `task_id` | **Y** | 고유 식별자 (예: `EPIC0-FE-001`) | 의존성 관리 및 작업 추적의 키로 사용 |
| `req_ids` | **Y** | 관련 SRS 요구사항 ID 목록 | 구현 시 해당 REQ의 제약조건을 우선적으로 고려 |
| `steps_hint` | N | 구현 단계에 대한 힌트 | 에이전트가 CoT(Chain of Thought)를 계획할 때 참고 |
| `inputs` / `outputs` | N | 입출력 데이터 명세 | 인터페이스 설계 및 테스트 케이스 생성의 기준 |
| `dependencies` | N | 선행 Task ID 목록 | 이 목록에 있는 Task가 완료되기 전에는 실행 대기 상태 유지 |
| `agent_profile` | N | 권장 에이전트 타입 | 작업 분배 시 사용 (`frontend`, `backend`, `infra` 등) |
| `example_commands` | N | 실행 프롬프트 예시 | 에이전트에게 작업을 지시할 때 사용할 수 있는 자연어 명령 |


## 3. 에이전트 오케스트레이션 전략

### 3.1. 작업 선택 및 순서 결정 (Scheduling)
1. **의존성 해결**: `dependencies` 필드에 명시된 Task들이 모두 완료(`completed`) 상태인지 확인합니다.
2. **병렬 실행**: `parallelizable: true`인 Task들은 의존성이 없다면 동시에 여러 에이전트 인스턴스에 할당할 수 있습니다.
3. **우선순위**: `priority` 필드(`Must` > `Should` > `Could`)와 `EPIC` 순서를 고려하여 먼저 실행할 작업을 큐에 넣습니다.

### 3.2. 실행 모드별 가이드

#### A. 단일 Task 실행 모드
- **입력**: 특정 Task 파일 경로 (예: `tasks/functional/EPIC0-FE-001.md`)
- **행동**:
    1. 파일의 전체 내용을 읽습니다.
    2. YAML 블록의 `context`, `inputs`, `steps_hint`를 파싱합니다.
    3. Markdown 본문의 "상세 요구사항"을 정독하여 비즈니스 맥락을 이해합니다.
    4. 코드를 생성/수정하고, `postconditions`에 정의된 조건이 충족되었는지 스스로 검증합니다.

#### B. Epic 단위 실행 모드 (Batch)
- **입력**: 특정 Epic 태그 (예: `epic: "EPIC_0_FE_PROTOTYPE"`)
- **행동**:
    1. 해당 Epic 태그를 가진 모든 Task 파일을 스캔합니다.
    2. 내부 의존성(DAG)을 분석하여 실행 순서를 정렬합니다.
    3. 순서대로(또는 병렬로) Task를 실행하며 컨텍스트(공통 컴포넌트, 스타일 가이드 등)를 유지합니다.

## 4. Epic 0 (FE PoC) 실행 특이사항
- **목표**: 완벽한 구현보다는 "화면 흐름"과 "데이터 구조" 검증이 우선입니다.
- **API 처리**: 백엔드가 아직 없으므로, 모든 API 호출은 `mock` 데이터나 로컬 상태로 처리해야 합니다.
- **스타일**: 세부적인 CSS보다는 레이아웃 구조와 반응형 동작 여부에 집중합니다.

## 5. Task 상태 관리 및 피드백
- 에이전트는 작업 완료 후 Task 파일 하단(또는 별도 로그)에 실행 결과를 기록하는 것이 권장됩니다.
- 구현 중 발견된 SRS의 모호함이나 불가능한 요구사항은 `Risk/Issue` 섹션에 추가하여 사람에게 피드백해야 합니다.
